## Choosing proof scope

`next_mode` controls edit scope. `local` authorizes proof-body edits to the active node and new helper nodes inside the active node's support cone. It does not authorize edits to the active node's `.tex`, edits to any existing other node, signature changes, or addressing blockers on other nodes. A non-empty `task_blocker_ids` is illegal under `local` because blocker repair usually needs edits outside the active proof body; the exception is a single Soundness blocker on the active node when `must_close_active=true`, since closing the proof in Lean auto-clears that Soundness blocker within Local's scope.

The "coarse DAG" is the set of nodes that were present at the moment theorem-stating was approved and the run advanced into proof-formalization. Those nodes went through paper-faithfulness + correspondence approval and their signatures are protected. Helper nodes added later (during proof-formalization) are not in the coarse DAG and are not similarly protected. The exact coarse-DAG node list for this run is in the request summary's `coarse_dag_nodes` field — check it before choosing a mode.

Use `restructure` when the active proof burden needs nearby support edits, helper refactors, or local branch reshaping, but the work is still centered on the active node. Under `restructure` the worker may change the signature (hypotheses, return type) of the active node **only if the active node is NOT in the coarse DAG** — i.e. only for helpers introduced during proof-formalization.

Use `coarse_restructure` when the repair requires changing the signature of a coarse-DAG node, or when the protected package boundary otherwise needs to move: several coarse-DAG nodes need coordinated edits, or a covering node's correspondence must be reopened.

For `restructure` and `coarse_restructure`, choose `next_active` as the present scope anchor whose dependency envelope covers the nodes you authorize. `next_active` must be one of `kernel_hinted_next_active_nodes`; that set is already filtered to the active coarse-anchor cone (see the cone-clean fragment for the anchor mechanics). If no listed node has a wide enough envelope for the repair, switch anchor first via `next_active_coarse` in the same response. On retry reviews with a needed non-protected out-of-cone edit, use `global_repair_request`.

Protected semantic changes are exceptional. Leave `protected_semantic_change_node_ids` empty unless the protected node really cannot be preserved; if you name any protected node, the kernel will reissue Review for explicit confirmation, and any actual worker reopen will require verifier drain plus human reapproval.

## Choosing closure gates

`allow_new_obligations` controls whether newly introduced helpers may remain Lean-open. Set it to `false` when you want the worker to avoid creating new proof obligations; every new helper must then be Lean-closed with no `sorry`. Set it to `true` when honest single-burst decomposition may require new helpers with `sorry` and NL proofs. These helpers still must satisfy the scope rules for the chosen `next_mode`. After cycle 1, this does not permit new proof-bearing nodes with `SKETCH:` NL proofs; the worker must either write complete NL proofs expected to pass strict soundness verification or not create those nodes. `false` is the preferred working mode when feasible, so that the number of lean obligations is not needlessly increasing, and to avoid NL soundness work that can be avoided by closing helpers when they are initially created. You should not usually set this to `true` unless recent history shows other strategies are insufficient to make progress on your current target.

`must_close_active` controls whether this burst must finish the active node. Set it to `true` when the assignment is explicitly a closure attempt; the active node must then be Lean-closed with no `sorry`. Set it to `false` when useful partial progress or decomposition should be accepted.

`allow_new_obligations`=`false`, `must_close_active`=`true` is thus the most restrictive setting of these choices, suitable for nodes plausibly closeable in one burst. On the other end of the spectrum, `allow_new_obligations`=`true`, `must_close_active`=`false` should be used with extreme caution; we don't like creating new obligations while making no progress. Of the intermediate options: `allow_new_obligations`=`true`, `must_close_active`=`true` is a workflow that prefers a complete decomposition into unclosed helpers in one go. This only makes sense if it seems like all the necessary helpers are themselves complex enough that they will also need to be further decomposed. On the other hand, with `allow_new_obligations`=`false`, `must_close_active`=`false`, you can direct a worker to peel off requirements of a node's NL proof into already-closed helpers one requirement at a time. One advantage of this workflow is that the new helper nodes then won't need their own NL soundness work/checks. You should usually prefer this over `allow_new_obligations`=`true`, `must_close_active`=`true` when you can confidently identify at least one necessary (and substantive) helper that can plausibly be added in one go. One workflow that can be effective in this mode is: write a multipart unfinished proof of node A with sorrys in multiple places, and then replace these sorrys with closed helpers one burst at a time.

New obligations are particularly dangerous when working in an area of the DAG where you are not confident that the existing statement scaffolding is rock-solid. You want to repair the existing structure before building on top of and around it.
